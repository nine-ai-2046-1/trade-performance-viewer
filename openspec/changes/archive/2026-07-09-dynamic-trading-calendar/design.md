## Context

現有嘅 index.html 使用硬編碼嘅 EVENTS 和 MONTHS 數據來渲染日曆。用戶需要一個動態嘅交易日曆，可以從 CSV 文件加載真實嘅交易數據，並在日曆上顯示每日每產品嘅 P/L 情況。

**當前狀態**:
- index.html 包含硬編碼嘅 events 數組
- 日曆只顯示固定嘅兩個月份（June 和 July 2026）
- 無數據加載或解析功能
- 無月份導航功能

**數據來源**:
- data/data.csv 包含交易記錄
- 日期範圍：2026-05-29 至 2026-07-09
- 產品：JPN225、US500、HK50
- 每筆交易包含：時間、產品、方向、建倉價、平倉價、數量、淨盈利

## Goals / Non-Goals

**Goals**:
- 實時從 CSV 加載交易數據（唔缓存）
- 計算每日每產品嘅 P/L 總和
- 計算每日總 P/L
- 根據數據範圍動態生成月份
- 支持月份導航（Prev/Next）及邊界檢查
- 在日曆格仔顯示產品名稱 + P/L 總和
- 使用 emoji 和顏色區分盈虧和產品

**Non-Goals**:
- 唔需要支持其他數據格式（只支持 CSV）
- 唔需要後端服務（純前端實現）
- 唔需要交易細節顯示（只顯示產品級別聚合）
- 唔需要響應式設計（保持現有佈局）

## Decisions

### 1. 數據加載方式
**選擇**: 使用 Fetch API 加載 CSV 文件
**理由**: 
- 純前端實現，無需後端
- 現代瀏覽器普遍支持
- 可以輕鬆添加 cache-control 標頭禁止缓存

**替代方案**: 
- 使用 XMLHttpRequest（較舊，代碼較多）
- 使用 Import Maps（兼容性較差）

### 2. CSV 解析方式
**選擇**: 自己編寫簡單嘅 CSV 解析函數
**理由**:
- CSV 格式簡單，無需引入外部庫
- 數據結構固定，可以針對性解析
- 避免外部依賴

**替代方案**:
- 使用 Papa Parse（引入不必要嘅依賴）

### 3. 數據存儲結構
**選擇**: 使用 Map 存儲每日每產品嘅 P/L
```
{
  "2026-07-09": {
    "JPN225": 1953.24,
    "US500": -892.15,
    "HK50": 578.90,
    "total": 1540.00
  }
}
```
**理由**:
- 按日期快速查找
- 易於計算和顯示
- 結構清晰

### 4. 月份導航邏輯
**選擇**: 從數據中提取可用月份範圍，動態生成月份列表
**理由**:
- 根據實際數據顯示月份
- 邊界檢查簡單（只顯示有數據嘅月份）
- 用戶體驗更好

### 5. 產品顏色方案
**選擇**: 為每個產品分配固定嘅 font color
- JPN225: #8B5CF6（紫色）
- US500: #3B82F6（藍色）
- HK50: #92400E（棕色）

**理由**:
- 顏色區分清晰
- 與背景對比度足夠
- 符合用戶要求

## Risks / Trade-offs

### 風險 1: CSV 格式變化
**風險**: 如果 CSV 格式變化，解析可能失敗
**缓解**: 在解析函數中添加錯誤處理，提供友好嘅錯誤提示

### 風險 2: 大數據量性能
**風險**: 如果交易記錄很多，加載和解析可能較慢
**缓解**: 數據量目前較小（<500行），可以接受。如果未來數據量增大，可以考慮分頁加載

### 風險 3: 瀏覽器兼容性
**風險**: 舊版瀏覽器可能不支持 Fetch API
**缓解**: 目標用戶使用現代瀏覽器，可以接受

### 權衡 1: 代碼複雜度 vs 功能
**權衡**: 自己編寫 CSV 解析增加了代碼量，但避免了外部依賴
**決定**: 選擇自己編寫，因為格式簡單且可控制

### 權衡 2: 實時加載 vs 性能
**權衡**: 每次頁面加載都 fetch CSV，可能影響性能
**決定**: 選擇實時加載以確保數據最新，因為數據量小且用戶要求唔缓存
