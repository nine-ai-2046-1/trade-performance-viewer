## Context

現有嘅日曆 P/L 顯示格式存在以下問題：
- 使用中文「總計」顯示總和
- 總計行無 emoji 邏輯
- 數值前無 $ 符號
- 數值包含小數點
- 月份標籤下方無月度 P/L 總和

**當前顯示格式**:
```
┌─────────────────────┐
│ 9                   │
│ 🟢 JPN225 +1953.24  │
│ 總計: +1953.24      │
└─────────────────────┘
```

**目標顯示格式**:
```
┌─────────────────────────────┐
│        June 2026            │
│   🟢 Total: +$12,345        │
└─────────────────────────────┘

┌─────────────────────┐
│ 9                   │
│ 🟢 JPN225 +$1953    │
│ 🟢 Total: +$1953    │
└─────────────────────┘
```

## Goals / Non-Goals

**Goals**:
- 移除小數點，使用整數顯示
- 所有 P/L 數值前加 $ 符號
- 總計行使用英文 "Total"
- 總計行加入 emoji 邏輯
- 月份標籤下方顯示月度 P/L 總和

**Non-Goals**:
- 唔修改日曆格仔佈局
- 唔修改產品顏色方案
- 唔修改月份導航功能

## Decisions

### 1. 數值格式化
**選擇**: 使用 `Math.round()` 取整，前加 $ 符號
**理由**:
- 用戶要求無小數點
- $ 符號增加可讀性

### 2. 月份 P/L 計算
**選擇**: 在 `renderMonth()` 中遍歷該月所有日期，計算總和
**理由**:
- 直接從 calendarData 中提取
- 無需新增數據結構

### 3. 顯示位置
**選擇**: 月份 P/L 顯示喺月份標籤下方
**理由**:
- 用戶明確要求放下一行
- 與月份標籤視覺分離

## Risks / Trade-offs

### 風險 1: 數值精度損失
**風險**: 取整後可能丟失精度
**缓解**: 交易 P/L 通常為整數，影響不大

### 風險 2: 月份 P/L 計算性能
**風險**: 每次渲染都需遍歷該月日期
**缓解**: 數據量小，性能影響可忽略
